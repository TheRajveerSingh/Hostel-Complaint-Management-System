# Sprint Retrospective - Hostel Complaint Management System
**Project Phase:** Operational Enhancements & UI Refinement
**Date:** April 30, 2026

| What went Well | What went Poorly | What ideas do we have | How should we take action |
| :--- | :--- | :--- | :--- |
| **Deployment Success:** Render deployment with Blueprint config is stable and consistent. | **Env Config Issues:** Initial deployment failed due to missing Supabase secrets in Render environment. | **Staging Environment:** Create a separate staging branch to test deployments before production. | Document all required keys in a `.env.example` file and verify during CI. |
| **Photo Evidence:** Camera integration for staff improved accountability for resolved complaints. | **Mobile Camera Bugs:** Camera capture UI had alignment issues on smaller mobile screens. | **Gallery Fallback:** Always allow file uploads as a fallback if the camera API fails. | Standardize camera component layout using responsive CSS flexbox. |
| **Data Analytics:** Warden dashboard graphs provide clear visibility into incident trends. | **Timezone Mismatch:** Incident velocity was grouped by UTC, causing data offsets in local reports. | **Trend Analysis:** Add a "Week-over-Week" comparison feature for the Warden. | Refactor date-grouping logic to strictly use the user's local timezone strings. |
| **Security Foundation:** Supabase RLS policies effectively isolate student/staff data. | **Policy Oversights:** Restrictive RLS initially blocked staff from seeing their assigned tasks. | **Security Audit Script:** Create a test suite to verify RLS policies for every user role. | Perform a full policy audit whenever a new database table or relation is added. |
| **Feedback UI:** 2-column layout for resolved issues improved the readability of long logs. | **UI Overcrowding:** Long feedback entries were previously causing excessive page scrolling. | **Accordion Views:** Use collapsible sections for historical feedback to save vertical space. | Apply the new 2-column layout pattern to all data-heavy detail views. |
| **Complaint Lifecycle:** Emergency prioritization ensures critical issues are seen first. | **SLA Awareness:** "General" category complaints lacked clear resolution deadlines. | **Automated Escalation:** System should auto-escalate complaints unaddressed for >48 hours. | Add color-coded "Time-Since-Created" badges to the Warden's queue. |
| **Dev Environment:** Vite + React allowed for extremely fast iteration and hot-reloading. | **Dependency Conflicts:** Occasional `package-lock.json` issues during collaborative updates. | **Standardized Tooling:** Enforce a specific Node/NPM version for all contributors. | Add a "Development Workflow" section to the README for onboarding. |
| **Feature Planning:** Using stepwise implementation plans kept the focus on core requirements. | **Ambiguity:** Some UI requirements were clarified only late in the implementation phase. | **UI Mockups First:** Generate visual mockups/wireframes before building complex views. | Include a "Visual Design" step in all future implementation plans. |
| **Auth Logic:** Robust multi-role login system (Student/Warden/Staff) is working perfectly. | **Redirect Edge Cases:** Deep-linking to specific complaints often lost auth context. | **Persistent Sessions:** Improve the "Remember Me" logic for maintenance staff on mobile. | Update the private route wrapper to capture and restore the `redirectUrl`. |
| **Staff Management:** Wardens can now easily manage the maintenance team via the portal. | **List Scalability:** The staff list becomes cluttered when managing a large team (>10). | **Search & Filter:** Add a search bar to the StaffList to find specific personnel quickly. | Implement basic filtering and pagination for the StaffList component. |
